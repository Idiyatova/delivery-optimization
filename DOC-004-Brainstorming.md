**Проект:** Оптимизация системы доставки  
**ID документа:** DOC-004-Brainstorming 
**Версия:** 1.0  

**Связанные документы:** [DOC-001-Preparation](pDOC-001-Preparation.md), [DOC-002-Needs](DOC-002-Needsmd), [DOC-003-RoleGame](DOC-003-RoleGame.md), [DOC-00-AsIs](DOC-005-AsIs.md)  

## 1. Введение  
Данный документ документирует результаты мозгового штурма для генерации идей решений, основанных на выводах ролевой игры ([DOC-003-RoleGame](DOC-003-RoleGame.md)) и потребностях стейкхолдеров ([DOC-002-Needs](needs.md)). Цель — предложить решения для текущих проблем, которые были проанализированы в [DOC-005-AsIs](DOC-005-AsIs.md).  

- **Методология:** Групповое обсуждение с фиксацией идей делопроизводителем.  

## 2. Проведение мозгового штурма  

### 2.1 Предложения по формированию бизнес-требований  

**Цель:** Сформулировать бизнес-требования, определяющие преимущества системы для стейкхолдеров, определить критерии успеха и определить границы проекта.  

| Проблема                                      | Предложения и решения                                                                 |
|-----------------------------------------------|---------------------------------------------------------------------------------------|
| 1. Какие ключевые проблемы стейкхолдеров решает система? | - Для клиентов: Реализовать единую платформу с трекингом заказов в реальном времени для прозрачности доставки, с уведомлениями о доставке. <br> - Для курьеров: Прозрачная система распределения заказов, автоматизация распределения заказов, возможность отменить заказ. <br> - Для операторов: Автоматизировать ввод заказов с шаблонами и валидацией для снижения ошибок. <br> - Для поставщиков: Обеспечить интеграцию с CRM поставщиков для упрощения взаимодействия и передачи заявки на заказ. |
| 2. Какие возможности системы обеспечат преимущества для стейкхолдеров? | - Единая платформа с трекингом - прозрачность для всех участников. <br> - Мобильные приложения (iOS/Android) для курьеров и веб-интерфейс для операторов - доступность с любых устройств. <br> - Интеграция с картографическими сервисами (Яндекс.Карты, 2GIS) - оптимизация маршрутов. <br> - Автоматизация ручных процессов - снижение операционных затрат. <br> - Система рейтингов и отзывов для контроля качества. |
| 3. Какие основные бизнес-цели определяют требования? | - Прибыльность: Снижение операционных затрат за счет автоматизации рутинных процессов. <br> - Привлечение клиентов: Гарантированная доставка в течение 30-60 минут как конкурентное преимущество. <br> - Соблюдение норм: Валидация данных и интеграция с проверенными сервисами для соответствия требованиям. |
| 4. Какой функционал MVP необходим на первом этапе? | - Базовый веб-интерфейс для операторов и диспетчеров - основа управления. <br> - Мобильное приложение для курьеров с функциями просмотра, бронирования заказов и уведомлений. <br> - Система уведомлений для клиентов о статусе доставки. <br> - Ежедневная финансовая отчетность для контроля показателей. |
| 5. По каким критериям определить, что система успешна? | - 95% доставок в срок до 60 минут - показатель эффективности логистики. <br> - Снижение операционных затрат за счет автоматизации рутинных процессов. <br> - Увеличение среднего количества заказов на курьера на 30% - рост производительности. |
| 6. Какие функции ключевых ролей должны быть реализованы? | - Курьер: Просмотр заказов, навигация, изменение статусов, экстренная связь с диспетчером. <br> - Диспетчер: Управление заказами, просмотр карты с курьерами, переназначение заказов, базовая аналитика, чат с курьерами. <br> - Оператор: Ввод заказов, использование шаблонов, валидация данных, история операций, поиск по заказам. <br> - Клиент: Отслеживание заказа, уведомления, история заказов, чат с поддержкой. |
| 7. Какие основные бизнес-цели и иерархия задач? | - Создать прибыльный сервис доставки. <br> - Иерархия задач: <br> - Привлечение 100+ постоянных клиентов в первый месяц. <br> - Снижение времени обработки заказа. |

### 2.2 Предложения по ролям пользователей системы, их проблемы as is  

**Цель:** Выявить текущие проблемы всех ролей пользователей системы  

| Роль                        | Проблема as is                                                                 |
|-----------------------------|--------------------------------------------------------------------------------|
| Клиент (потребитель)        | - Не знает точное время доставки и текущий статус заказа. <br> - Не может оперативно связаться с курьером при изменении обстоятельств. <br> - Сталкивается с поврежденными товарами при доставке. <br> - Беспокоится о сохранности персональных данных. |
| Курьер                      | - Приложение часто зависает или теряет соединение. <br> - Не видит актуальный список доступных заказов в реальном времени. <br> - Непрозрачная система расчета оплаты и задержки выплат. <br> - Отсутствие удобной навигации и информации о парковках. |
| Оператор                    | - Поставщики присылают данные в разных форматах (мессенджеры, email, телефон). <br> - Необходимость ручного ввода приводит к ошибкам и перегрузке. <br> - Нет автоматической проверки корректности адресов и контактов. <br> - Трудно найти историю заказов при обращении клиентов. |
| Диспетчер                   | - Не видит реальное местоположение курьеров на карте. <br> - Система не показывает актуальные статусы заказов. <br> - Нет инструментов для быстрого переназначения заказов. <br> - Приходится решать конфликты с клиентами из-за задержек. <br> - Координация работы курьеров по телефону трудозатратна. |
| Бухгалтерия                 | - Данные о доставках приходят с ошибками и неполные. <br> - Много ручной работы при переносе данных в бухгалтерскую систему. <br> - Несовместимость форматов выгрузки с используемыми системами. <br> - Задержки в расчетах с курьерами и поставщиками. |
| Поставщик                    | - Курьеры долго забирают заказы или доставляют с опозданием. <br> - Нет уведомлений о статусе доставки и прибытии курьера. <br> - Ошибки курьеров при передаче заказов. <br> - Сложности с финансовыми отчетами и расчетами. |
| Администратор системы       | - Нет централизованного управления доступом. <br> - Нет инструментов мониторинга работоспособности системы. |
| Руководитель системы доставки | - Задержка в передаче информации. <br> - Высокая вероятность потери/искажения данных. <br> - Высокие операционные затраты из-за дублирования функций и ручных процессов. <br> - Невозможность оперативно получать аналитику по ключевым бизнес-показателям. |
| Разработчики онлайн-системы  | - Пользователи часто пишут неконструктивную критику. <br> - Частые изменения требований в процессе разработки. <br> - Отсутствие обратной связи о качестве исправлений. |

## 5. Приоритизация предложений  
На основе анализа проблем и решений из 2.1 и 2.2, предложения приоритизированы по критериям: влияние на ключевых стейкхолдеров (клиенты, курьеры), стоимость реализации и соответствие MVP.  

| Приоритет | Предложение / Решение                                                                 | Обоснование |
|-----------|---------------------------------------------------------------------------------------|-------------|
| Высокий   | Реализовать единую платформу с трекингом заказов в реальном времени для клиентов    | Увеличивает прозрачность и удовлетворенность клиентов, ключевой фактор привлечения. |
| Высокий   | Автоматизировать распределение заказов для курьеров                                   | Снижает задержки, повышает производительность, критично для операционной эффективности. |
| Высокий   | Интеграция с картографическими сервисами для оптимизации маршрутов                    | Решает проблему навигации, снижает время доставки, подходит для MVP. |
| Высокий   | Автоматизировать ввод заказов с шаблонами для операторов                              | Снижает ошибки и операционные затраты, улучшает скорость обработки. |
| Средний   | Система уведомлений для клиентов о статусе доставки                                   | Улучшает пользовательский опыт, но зависит от базовой интеграции. |
| Средний   | Система рейтингов и отзывов для контроля качества                                     | Повышает доверие, но может быть добавлена после MVP. |
| Средний   | Интеграция с CRM поставщиков                                                          | Упрощает взаимодействие, но требует дополнительных ресурсов. |
| Низкий    | Расширенная аналитика для руководителя                                                | Полезно для масштабирования, но не критично для MVP. |
